Задание 3

Это задание описал в виде шагов , но позже немного изменил структуру повествования .

1. Настройка компиляторов 
Мы начали сборку с windows. Оказалось, что стандартных расширений VS Code недостаточно, 
поэтому была установлена Visual Studio 2022 для использования компилятора MSVC.
Для сборки под Linux использовалась подсистема WSL, интегрированная в VS Code, 
что позволило нам работать в одном редакторе.
Также пришлось разобраться с Cmaketools .

2. Сборка и решение проблем с путями 
Изначально пробовал собрать Cmakelist через static ,это не подходило по условию задания ,мы перешли на shared.
После сборки под Linux исполняемый файл не мог найти библиотеку без ручной настройки путей.
В CMakelists был внедрен механизм RPATH через параметры INSTALL_RPATH "$ORIGIN" и BUILD_WITH_INSTALL_RPATH TRUE. 
Это позволило программе искать .so файл в той же директории, где и она сама находится.

3. Тестирование
Для подтверждения работоспособности библиотек был создан тестовый файл .c. 
Программа успешно обращалась к собранной библиотеке и выводила версию SQLite, т.е. все работало , как надо.

4. Вывод логов
При генерации логов на Windows у нас возникла проблема с кодировкой текста , связанная с особенностями MSBuild.
Мы перевели лог на английский язык и сократили его с помощью флага /v:m (minimal verbosity).
У нас получился вывод лога , как на linux-e.

Задание 4 

Создали dockerfile.
В dockerfile указываем alpine linux , т.к. требуется создать легковесное окружение .

В строке CMD ["gcc", "--version"] , написал --version , для того чтобы выводилась версия при обычном docker run-e .
(Эта информация показывает нам, что комплиятор срабатывает)

Собрали образ , docker build -t sqlite-builder 

Проверили его наличие через docker images

Запускаемся командой docker run --rm -v $(pwd):/app sqlite-builder gcc -shared -fPIC sqlite3.c -o libsqlite3.so -lpthread -ldl 2>&1 | tee dockerlog_linux.txt

Здесь мы используем флаги -shared для того, чтобы получить файл .so (Т.е. его можно подключить другой программой)
-fPIC делает код совместимым с любым участком оперативной памяти, который ему выделит ему система при запуске, 
 если честно не до коцна разобрался .

lpthread включает многопоточность , она поддерживается в sqlite
ldl Подключает библиотеку для динамической загрузки Флаг -ldl дает библиотеке инструменты для того, чтобы находить и открывать другие файлы в системе.
2>&1 | tee используется для одновременного вывода процесса компиляции в терминал и записи полного лога. 
В этой записи:
stdout (канал 1) — сюда идет обычный текст (успешные сообщения).
stderr (канал 2) — сюда идут только ошибки
tee азделяет поток данных на два направления (В консоль и в файл).

Лог был пустым, значит никаких ошибок у нас не было ,продублировал туда комаду.
И у нас появился libsqlite3.so в нашей папке.

Vagrantfile
Сначала запускается vargant file , который в свою очередь создает ВМ(виртуальную машину) на базе ос Debian и настраивает ее.
Ос будет загружаться из зеркала , поскольку есть проблема с ее загрузкой из официального репозитория.
Дальше синхронизируются папки , решается проблема с репозиторием backports , 
помимо этого решает проблему с тем , что время на ВМ может не совпадать с нашим ,
что приводило к ошибке . В конце загружается ansible и вся работа дальше будет проходить в нем.

dockerplaybook.yml

Описание dockerplaybook.yml
У нас есть 3 тэга для заданий ,которые разделены логически и в каждом из них выполняется какое-то задание ,по стандарту они выполнятся по порядку.
Тэги:

install_docker который устанавливает на созданную нами виртуальную машину пакет Docker из репозитория
Sqlite который дублирует задания 1-4 загружает Sqlite и производит его сборку с использованием docker
Conan На котором выполняется задание 8 с использованием conan соответсвенно 

Если есть желание проверить работу какого-то конкретного тэга 
ansible-playbook dockerplaybook.yml --tags Sqlite --list-tasks 
для примера здесь проверяется tag Sqlite или можно запустить все кроме какого-то конкретного тэга
